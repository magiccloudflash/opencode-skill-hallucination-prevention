## Χαρακτηριστικά

- **Επαλήθευση πριν από δήλωση** — διάβασμα αρχείων, εκτέλεση κώδικα, έλεγχος τεκμηρίωσης πριν από ισχυρισμούς
- **Αναγνώριση αβεβαιότητας** — σαφής επισήμανση εικασίας έναντι επαληθευμένων γεγονότων
- **Μην επινοείτε** — ποτέ μην κατασκευάζετε σφάλματα, API, αποτελέσματα δοκιμών ή διορθώσεις
- **Τεκμηρίωση σε αποδεικτικά στοιχεία** — αναφορά διαδρομών αρχείων, αριθμών γραμμών και πραγματικής εξόδου
- **Ροή αυτοελέγχου** — λίστα ελέγχου 5 βημάτων πριν από κάθε απόκριση
- **Προστατευτικά κιγκλιδώματα δημιουργίας κώδικα** — επαλήθευση πριν/μετά τη γραφή
- **Πρωτόκολλο διαχείρισης σφαλμάτων** — ανάγνωση πραγματικών σφαλμάτων, εμφάνιση αυτολεξεί, επαλήθευση διορθώσεων

## Εγκατάσταση

### Επιλογή 1: Κλωνοποίηση αυτού του αποθετηρίου

```bash
git clone https://github.com/<your-org>/opencode-skill-hallucination-prevention.git
```

Στη συνέχεια, αντιγράψτε τη δεξιότητα στην καθολική διαμόρφωση opencode:

```bash
# macOS / Linux
cp -r hallucination-prevention ~/.config/opencode/skills/

# Windows PowerShell
Copy-Item -Recurse hallucination-prevention "$env:USERPROFILE\.config\opencode\skills\"
```

### Επιλογή 2: Μη αυτόματη εγκατάσταση

Δημιουργήστε τον κατάλογο δεξιότητας και αντιγράψτε το `SKILL.md`:

```bash
# macOS / Linux
mkdir -p ~/.config/opencode/skills/hallucination-prevention

# Windows PowerShell
mkdir -p "$env:USERPROFILE\.config\opencode\skills\hallucination-prevention"
```

Στη συνέχεια, τοποθετήστε το [`hallucination-prevention/SKILL.md`](hallucination-prevention/SKILL.md) σε αυτόν τον κατάλογο.

### Επιλογή 3: Ως καθολικός κανόνας (αυτόματη φόρτωση σε κάθε συνεδρία)

Αντιγράψτε τους βελτιωμένους κανόνες στο καθολικό σας AGENTS.md:

```bash
# macOS / Linux
cat hallucination-prevention/SKILL.md >> ~/.config/opencode/AGENTS.md

# Windows PowerShell
Add-Content -Path "$env:USERPROFILE\.config\opencode\AGENTS.md" -Value (Get-Content "hallucination-prevention\SKILL.md")
```

Ή χρησιμοποιήστε το αυτόνομο [`AGENTS.md`](AGENTS.md) που παρέχεται σε αυτό το αποθετήριο.

## Χρήση

Η δεξιότητα ανακαλύπτεται αυτόματα από το OpenCode. Ο πράκτορας θα τη βλέπει στη λίστα διαθέσιμων δεξιοτήτων και θα τη φορτώνει όταν είναι σχετική.

Για ρητή φόρτωση κατά τη διάρκεια μιας συνεδρίας:

```
Ο πράκτορας μπορεί να χρησιμοποιήσει: skill({ name: "hallucination-prevention" })
```

### Διαμόρφωση αδειών

Στο `opencode.json` σας, μπορείτε να ελέγχετε την πρόσβαση:

```json
{
  "permission": {
    "skill": {
      "hallucination-prevention": "allow"
    }
  }
}
```

Για παράκαμψη ανά πράκτορα:

```json
{
  "agent": {
    "plan": {
      "permission": {
        "skill": {
          "hallucination-prevention": "allow"
        }
      }
    }
  }
}
```

## Κανόνες ανά εργαλείο

### read — ΠΟΤΕ μην μαντεύετε περιεχόμενο αρχείων. Επαληθεύστε ότι τα αρχεία υπάρχουν πριν τα διαβάσετε. Ξαναδιαβάστε μετά από επεξεργασίες.
### write — Πάντα ξαναδιαβάζετε αρχεία που γράψατε. Μην δημιουργείτε τεκμηρίωση χωρίς να σας ζητηθεί.
### edit — ΠΑΝΤΑ διαβάζετε πριν από επεξεργασία. Ξαναδιαβάζετε μετά από επεξεργασία για επιβεβαίωση. Αν η επεξεργασία αποτύχει, ξαναδιαβάστε για να βρείτε το πραγματικό κείμενο.
### bash — ΠΟΤΕ μην φαντάζεστε έξοδο εντολών. Εκτελέστε εντολές για να δείτε πραγματική έξοδο. Μην ισχυρίζεστε ότι μια διόρθωση λειτουργεί χωρίς επανέλεγχο.
### grep — Χρησιμοποιήστε για να επιβεβαιώσετε την ύπαρξη συναρτήσεων/κλάσεων πριν αναφερθείτε σε αυτές.
### glob — Χρησιμοποιήστε για να ανακαλύψετε την πραγματική δομή αρχείων πριν αναφερθείτε σε διαδρομές.
### question — Όταν οι οδηγίες είναι ασαφείς, ρωτήστε ΠΡΙΝ ενεργήσετε. Προσφέρετε συγκεκριμένες επιλογές.

## Κανόνες Git

- ΠΟΤΕ μην πλαστογραφείτε hashes commits — `git log` για πραγματικά
- ΠΟΤΕ μην υποθέτετε μηνύματα commits — `git log --oneline` για πραγματικά
- ΠΟΤΕ μην επινοείτε αριθμούς PR ή αναγνωριστικά issues
- Μετά από ένα commit, εκτελέστε `git status` και `git log -1` για επαλήθευση
- Εκτελέστε `git branch -a` πριν ισχυριστείτε ότι υπάρχουν branches

## Κανόνες γλώσσας/πλαισίου

- **JS/TS**: Ελέγξτε το `package.json` πριν από εισαγωγή. Ελέγξτε το `tsconfig.json` πριν ισχυριστείτε ρυθμίσεις.
- **Python**: Ελέγξτε το `requirements.txt`/`pyproject.toml` πριν από εισαγωγή. Επαληθεύστε `python --version`.
- **Rust**: Ελέγξτε το `Cargo.toml` για εξαρτήσεις/χαρακτηριστικά. Επαληθεύστε `rust-toolchain.toml`.
- **Go**: Ελέγξτε το `go.mod` για πληροφορίες ενότητας. Επαληθεύστε `go version`.
- **Shell**: Επαληθεύστε τις διαθέσιμες εντολές πριν από scripting. Δοκιμάστε one-liners πριν προτείνετε.

## Κανόνες URL/αναφοράς

- ΠΟΤΕ μην δημιουργείτε ένα URL που δεν έχετε επισκεφθεί ή επαληθεύσει
- Για τεκμηρίωση, προτιμήστε πραγματικές διαδρομές αρχείων στο αποθετήριο
- Αν δεν είστε σίγουροι για ένα URL: ρωτήστε τον χρήστη ή δηλώστε "not verified"

## Πρωτόκολλο κλιμάκωσης — Σταματήστε και ρωτήστε όταν:

1. Οι οδηγίες είναι ασαφείς και ανοιχτές σε πολλαπλές ερμηνείες

## Βασικές αρχές

### 1. Επαλήθευση πριν από δήλωση
- Διαβάστε αρχεία πριν ισχυριστείτε περιεχόμενο. Εκτελέστε κώδικα πριν αναφέρετε αποτελέσματα.
- Επιβεβαιώστε την ύπαρξη συναρτήσεων/κλάσεων με grep πριν αναφερθείτε.
- Ποτέ μην υποθέτετε ότι υπάρχουν διαδρομές αρχείων, υπογραφές ή εισαγωγές.

### 2. Αναγνώριση αβεβαιότητας
- Πείτε "δεν γνωρίζω" όταν οι πληροφορίες δεν είναι διαθέσιμες.
- Επισημάνετε την εικασία σαφώς: "πιθανώς", "μάλλον", "φαίνεται".
- Διακρίνετε το παρατηρημένο γεγονός από το συμπέρασμα.

### 3. Μην επινοείτε
- Ποτέ μην κατασκευάζετε: μηνύματα σφάλματος, έξοδο καταγραφής, μεθόδους API, επιλογές διαμόρφωσης, αποτελέσματα δοκιμών, hashes commits, αριθμούς PR, URL, έξοδο εντολών.
- Ποτέ μην ισχυρίζεστε ότι μια διόρθωση λειτουργεί χωρίς να την εκτελέσετε.

### 4. Τεκμηρίωση σε αποδεικτικά στοιχεία
- Αναφέρετε διαδρομές αρχείων + αριθμούς γραμμών. Παραθέστε πραγματικό κώδικα. Αναφερθείτε σε πραγματική έξοδο εντολών. Εμφανίστε πραγματικές διαφορές.

### 5. Επικύρωση υποθέσεων
- Καταγράψτε κατάλογο πριν υποθέσετε δομή. Διαβάστε πηγή πριν υποθέσετε υπογραφές.
- Ελέγξτε το package.json πριν υποθέσετε εξαρτήσεις. Διαβάστε διαμόρφωση πριν υποθέσετε ρυθμίσεις.
- Ξαναδιαβάστε τροποποιημένα αρχεία για επιβεβαίωση εφαρμογής επεξεργασιών.

## Προστατευτικά κιγκλιδώματα δημιουργίας κώδικα

### Πριν από τη γραφή κώδικα
1. Επαληθεύστε ότι οι εισαγωγές υπάρχουν στο package.json
2. Επαληθεύστε υπογραφές συναρτήσεων από πραγματικούς ορισμούς τύπων
3. Καταγράψτε τον κατάλογο προορισμού πριν δημιουργήσετε νέα αρχεία
4. Διαβάστε υπάρχοντα αρχεία για να ταιριάξετε συμβάσεις και στυλ
5. Επαληθεύστε ότι το πλαίσιο χρησιμοποιείται πραγματικά
6. Ελέγξτε την έκδοση γλώσσας πριν χρησιμοποιήσετε νέες δυνατότητες

### Μετά από τη γραφή κώδικα
1. Ξαναδιαβάστε το γραμμένο αρχείο για επιβεβαίωση ορθότητας
2. Εκτελέστε linter για επαλήθευση απουσίας συντακτικών σφαλμάτων
3. Εκτελέστε έλεγχο τύπων αν είναι διαθέσιμος
4. Εκτελέστε σχετικές δοκιμές
5. Αναφέρετε πραγματική επιτυχία/αποτυχία — εμφανίστε πραγματική έξοδο
6. Εμφανίστε διαφορά για έλεγχο από τον χρήστη

5. ΖΗΤΗΣΤΕ λεπτομέρειες — αν η έξοδος είναι περικομμένη
6. ΔΟΚΙΜΑΣΤΕ τη διόρθωση — εκτελέστε την ίδια εντολή για απόδειξη επιτυχίας
7. Σε αποτυχία — εμφανίστε το νέο σφάλμα, όχι το παλιό
8. ΤΕΚΜΗΡΙΩΣΤΕ — ποια ήταν η αιτία και πώς διορθώθηκε

## Επίπεδα εμπιστοσύνης

- **Επαληθευμένο**: Το διάβασα/εκτέλεσα/έλεγξα άμεσα
- **Πιθανό**: Ισχυρά στοιχεία αλλά όχι άμεσα επαληθευμένο
- **Εικαστικό**: Συμπέρασμα, χρειάζεται επαλήθευση
- **Άγνωστο**: Δεν μπορώ να προσδιορίσω από διαθέσιμες πληροφορίες

Προεπιλογή σε **Επαληθευμένο** — χρησιμοποιείτε πάντα εργαλεία για να φτάσετε σε αυτό το επίπεδο.

## Απαιτούμενες συμπεριφορές

- **Διάβασμα πρώτα, μετά γράψιμο**: Διαβάστε ένα αρχείο πριν το επεξεργαστείτε
- **Εκτέλεση για επιβεβαίωση**: Εκτελέστε κώδικα για επαλήθευση αλλαγών
- **Έλεγχος ύπαρξης**: Επαληθεύστε ότι διαδρομές, πακέτα, εργαλεία υπάρχουν
- **Αναφορά πραγματικών**: Εμφανίστε πραγματικές εξόδους, όχι φανταστικές
- **Σήμανση αβεβαιότητας**: Σημειώστε οτιδήποτε δεν είναι 100% σίγουρο
- **Επαλήθευση επεξεργασιών**: Ξαναδιαβάστε αρχεία μετά από επεξεργασία
- **Εμφάνιση διαφορών**: Παρουσιάστε αλλαγές για έλεγχο
- **Επαναγκύρωση περιβάλλοντος**: Ξαναδιαβάστε αρχεία μετά από 5+ κλήσεις εργαλείων
- **Έλεγχος αποκρίσεων**: Εκτελέστε τη λίστα ελέγχου 10 σημείων πριν από την παράδοση
- **Κλιμάκωση, μην μαντεύετε**: Στα όρια γνώσης, ρωτήστε τον χρήστη
