## คุณสมบัติ

- **ตรวจสอบก่อนกล่าวอ้าง** — อ่านไฟล์ รันโค้ด ตรวจสอบเอกสารก่อนทำการอ้างสิทธิ์
- **ยอมรับความไม่แน่นอน** — แยกแยะการคาดเดากับข้อเท็จจริงที่ตรวจสอบแล้วอย่างชัดเจน
- **อย่าสร้างข้อมูลเท็จ** — ห้ามสร้างข้อผิดพลาด, API, ผลการทดสอบ, หรือการแก้ไขที่ไม่มีจริง
- **ยึดหลักฐาน** — อ้างอิงพาธไฟล์ หมายเลขบรรทัด และผลลัพธ์จริง
- **ขั้นตอนตรวจสอบตนเอง** — รายการตรวจสอบ 5 ขั้นตอนก่อนทุกการตอบกลับ
- **การป้องกันการสร้างโค้ด** — การตรวจสอบก่อนเขียนและหลังเขียน
- **โปรโตคอลการจัดการข้อผิดพลาด** — อ่านข้อผิดพลาดจริง แสดงตามต้นฉบับ ตรวจสอบการแก้ไข

## การติดตั้ง

### ตัวเลือกที่ 1: โคลน repo นี้

```bash
git clone https://github.com/<your-org>/opencode-skill-hallucination-prevention.git
```

จากนั้นคัดลอกทักษะไปยังการกำหนดค่า opencode ส่วนกลางของคุณ:

```bash
# macOS / Linux
cp -r hallucination-prevention ~/.config/opencode/skills/

# Windows PowerShell
Copy-Item -Recurse hallucination-prevention "$env:USERPROFILE\.config\opencode\skills\"
```

### ตัวเลือกที่ 2: ติดตั้งด้วยตนเอง

สร้างไดเรกทอรีทักษะและคัดลอก `SKILL.md`:

```bash
# macOS / Linux
mkdir -p ~/.config/opencode/skills/hallucination-prevention

# Windows PowerShell
mkdir -p "$env:USERPROFILE\.config\opencode\skills\hallucination-prevention"
```

จากนั้นวาง [`hallucination-prevention/SKILL.md`](hallucination-prevention/SKILL.md) ลงในไดเรกทอรีนั้น

### ตัวเลือกที่ 3: เป็นกฎส่วนกลาง (โหลดอัตโนมัติทุกเซสชัน)

คัดลอกกฎที่ปรับปรุงแล้วไปยัง AGENTS.md ส่วนกลางของคุณ:

```bash
# macOS / Linux
cat hallucination-prevention/SKILL.md >> ~/.config/opencode/AGENTS.md

# Windows PowerShell
Add-Content -Path "$env:USERPROFILE\.config\opencode\AGENTS.md" -Value (Get-Content "hallucination-prevention\SKILL.md")
```

หรือใช้ [`AGENTS.md`](AGENTS.md) แบบสแตนด์อโลนที่ให้มาใน repo นี้

## การใช้งาน

ทักษะจะถูกค้นพบโดย OpenCode โดยอัตโนมัติ เอเจนต์จะเห็นทักษะที่พร้อมใช้งานและโหลดเมื่อเกี่ยวข้อง

เพื่อโหลดทักษะอย่างชัดเจนระหว่างเซสชัน:

```
The agent can use: skill({ name: "hallucination-prevention" })
```

### การกำหนดค่าสิทธิ์

ใน `opencode.json` ของคุณ คุณสามารถควบคุมการเข้าถึง:

```json
{
  "permission": {
    "skill": {
      "hallucination-prevention": "allow"
    }
  }
}
```

สำหรับการแทนที่แบบต่อเอเจนต์:

```json
{
  "agent": {
    "plan": {
      "permission": {
        "skill": {
          "hallucination-prevention": "allow"
        }
      }
    }
  }
}
```

### กฎเฉพาะเครื่องมือ

- **read** — ห้ามเดาเนื้อหาไฟล์ ตรวจสอบว่าไฟล์มีอยู่ก่อนอ่าน อ่านซ้ำหลังแก้ไข
- **write** — อ่านไฟล์ที่เขียนซ้ำเสมอ อย่าสร้างเอกสารโดยไม่ถูกขอ
- **edit** — อ่านก่อนแก้ไขเสมอ อ่านซ้ำหลังแก้ไขเพื่อยืนยัน ถ้าแก้ไขล้มเหลว ให้อ่านซ้ำเพื่อหาข้อความจริง
- **bash** — ห้ามสร้างผลลัพธ์คำสั่งที่ไม่มีจริง รันคำสั่งเพื่อดูผลลัพธ์จริง อย่าอ้างว่าการแก้ไขสำเร็จโดยไม่ทดสอบซ้ำ
- **grep** — ใช้เพื่อยืนยันการมีอยู่ของฟังก์ชัน/คลาสก่อนอ้างอิง
- **glob** — ใช้เพื่อค้นพบโครงสร้างไฟล์จริงก่อนอ้างอิงพาธ
- **question** — เมื่อคำแนะนำไม่ชัดเจน ให้ถามก่อนดำเนินการ เสนอตัวเลือกที่เป็นรูปธรรม

### โปรโตคอลการยึดติดบริบท

1. ก่อนดำเนินการกับไฟล์ครั้งใหญ่ ให้อ่านไฟล์ต้นฉบับที่เกี่ยวข้องซ้ำ
2. หลัง 5+ การเรียกใช้เครื่องมือ ให้อ่านไฟล์ที่แก้ไขซ้ำเพื่อยืนยันสถานะที่สะสม
3. เมื่อกลับมาทำงาน ให้อ่านไฟล์ที่เกี่ยวข้องทั้งหมดซ้ำ — อย่าเชื่อหน่วยความจำ

### กฎเฉพาะ Git

- ห้ามสร้าง hash คอมมิตที่ไม่มีจริง — ใช้ `git log` เพื่อหาค่าจริง
- ห้ามสมมติข้อความคอมมิต — ใช้ `git log --oneline` เพื่อหาข้อความจริง
- ห้ามสร้างหมายเลข PR หรือ issue ID
- หลังจากคอมมิต ให้รัน `git status` และ `git log -1` เพื่อยืนยัน
- รัน `git branch -a` ก่อนอ้างว่ามีสาขาอยู่

### กฎภาษา/เฟรมเวิร์ก

- **JS/TS**: ตรวจสอบ `package.json` ก่อนนำเข้า ตรวจสอบ `tsconfig.json` ก่อนอ้างถึงการตั้งค่า
- **Python**: ตรวจสอบ `requirements.txt`/`pyproject.toml` ก่อนนำเข้า ตรวจสอบ `python --version`
- **Rust**: ตรวจสอบ `Cargo.toml` สำหรับ deps/features ตรวจสอบ `rust-toolchain.toml`
- **Go**: ตรวจสอบ `go.mod` สำหรับข้อมูลโมดูล ตรวจสอบ `go version`
- **Shell**: ตรวจสอบคำสั่งที่พร้อมใช้งานก่อนเขียนสคริปต์ ทดสอบคำสั่งสั้น ๆ ก่อนแนะนำ

### กฎ URL/การอ้างอิง

- ห้ามสร้าง URL ที่คุณไม่ได้เยี่ยมชมหรือตรวจสอบ
- สำหรับเอกสาร ให้ใช้พาธไฟล์จริงใน repo
- ถ้าไม่แน่ใจเกี่ยวกับ URL: ให้ถามผู้ใช้ หรือระบุ "not verified"

### หลักการหลัก

#### 1. ตรวจสอบก่อนกล่าวอ้าง
- อ่านไฟล์ก่อนอ้างเนื้อหา รันโค้ดก่อนรายงานผลลัพธ์
- ยืนยันการมีอยู่ของฟังก์ชัน/คลาสด้วย grep ก่อนอ้างอิง
- ห้ามสมมติว่าพาธไฟล์ ลายเซ็น หรือการนำเข้ามีอยู่

#### 2. ยอมรับความไม่แน่นอน
- พูดว่า "ฉันไม่รู้" เมื่อไม่มีข้อมูล
- ติดป้ายการคาดเดาอย่างชัดเจน: "น่าจะ", "อาจจะ", "ดูเหมือนว่า"
- แยกแยะข้อเท็จจริงที่สังเกตได้จากการอนุมาน

#### 3. อย่าสร้างข้อมูลเท็จ
- ห้ามสร้าง: ข้อความข้อผิดพลาด, ผลลัพธ์ของ log, วิธีการ API, ตัวเลือก config, ผลการทดสอบ, hash คอมมิต, หมายเลข PR, URL, ผลลัพธ์คำสั่ง
- ห้ามอ้างว่าการแก้ไขสำเร็จโดยไม่รันจริง

#### 4. ยึดหลักฐาน
- อ้างอิงพาธไฟล์ + หมายเลขบรรทัด อ้างอิงโค้ดจริง อ้างอิงผลลัพธ์คำสั่งจริง แสดง diff จริง

#### 5. ตรวจสอบสมมติฐาน
- แสดงรายการไดเรกทอรีก่อนสมมติโครงสร้าง อ่านแหล่งที่มาก่อนสมมติลายเซ็น
- ตรวจสอบ package.json ก่อนสมมติ dependencies อ่าน config ก่อนสมมติการตั้งค่า
- อ่านไฟล์ที่แก้ไขซ้ำเพื่อยืนยันว่าการแก้ไขถูกนำไปใช้

### ระดับความเชื่อมั่น

- **Verified (ยืนยันแล้ว)**: ฉันอ่าน/รัน/ตรวจสอบโดยตรง
- **Likely (น่าจะ)**: หลักฐานชัดเจนแต่ไม่ได้ตรวจสอบโดยตรง
- **Speculative (คาดเดา)**: การอนุมาน ต้องการการตรวจสอบ
- **Unknown (ไม่ทราบ)**: ไม่สามารถระบุได้จากข้อมูลที่มีอยู่

ค่าเริ่มต้นคือ **Verified** — ใช้เครื่องมือเสมอเพื่อไปให้ถึงระดับนี้

### พฤติกรรมที่จำเป็น

- **อ่านก่อน แล้วค่อยเขียน**: อ่านไฟล์ก่อนแก้ไข
- **รันเพื่อยืนยัน**: รันโค้ดเพื่อตรวจสอบการเปลี่ยนแปลง
- **ตรวจสอบการมีอยู่**: ตรวจสอบว่าพาธ, แพ็คเกจ, เครื่องมือมีอยู่
- **รายงานผลลัพธ์จริง**: แสดงผลลัพธ์จริง ไม่ใช่สิ่งที่จินตนาการ
- **ระบุความไม่แน่นอน**: ทำเครื่องหมายสิ่งที่เชื่อถือได้ไม่ 100%
- **ตรวจสอบการแก้ไข**: อ่านไฟล์ซ้ำหลังแก้ไข
- **แสดง diff**: นำเสนอการเปลี่ยนแปลงเพื่อตรวจสอบ
- **ยึดติดบริบทใหม่**: อ่านไฟล์ซ้ำหลัง 5+ การเรียกใช้เครื่องมือ
- **ตรวจสอบการตอบกลับ**: รันรายการตรวจสอบ 10 ข้อก่อนส่ง
- **ยกระดับ อย่าเดา**: ที่ขอบเขตความรู้ ให้ถามผู้ใช้
