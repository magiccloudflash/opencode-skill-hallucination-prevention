## विशेषताएँ

- **पुष्टि करने से पहले सत्यापित करें** — निष्कर्ष निकालने से पहले फ़ाइलें पढ़ें, कोड चलाएँ, दस्तावेज़ देखें
- **अनिश्चितता स्वीकार करें** — अटकलों और सत्यापित तथ्यों के बीच स्पष्ट रूप से अंतर करें
- **मन से न बनाएँ** — त्रुटियाँ, API, परीक्षण परिणाम या सुधार कभी न गढ़ें
- **साक्ष्य पर आधारित रहें** — फ़ाइल पथ, पंक्ति संख्याएँ और वास्तविक आउटपुट उद्धृत करें
- **आत्म-जाँच** — प्रत्येक उत्तर से पहले 10-बिंदु जाँच सूची
- **कोड निर्माण सुरक्षा** — कोड लिखने से पहले और बाद में दोहरी जाँच
- **उपकरण-विशिष्ट नियम** — read, write, edit, bash, grep, glob आदि के लिए विशिष्ट भ्रम-विरोधी नियम
- **8 प्रकार के भ्रम** — प्रत्येक प्रकार की अपनी रोकथाम रणनीति

## त्वरित स्थापना

### विधि 1: रिपॉजिटरी क्लोन करें (अनुशंसित)

```bash
git clone https://github.com/magiccloudflash/opencode-skill-hallucination-prevention.git
```

फिर कौशल को अपने वैश्विक OpenCode कॉन्फ़िगरेशन में कॉपी करें:

```bash
# macOS / Linux
cp -r hallucination-prevention ~/.config/opencode/skills/

# Windows PowerShell
Copy-Item -Recurse hallucination-prevention "$env:USERPROFILE\.config\opencode\skills\"
```

### विधि 2: मैन्युअल स्थापना

```bash
# macOS / Linux
mkdir -p ~/.config/opencode/skills/hallucination-prevention
cp hallucination-prevention/SKILL.md ~/.config/opencode/skills/hallucination-prevention/

# Windows PowerShell
mkdir -p "$env:USERPROFILE\.config\opencode\skills\hallucination-prevention"
Copy-Item hallucination-prevention/SKILL.md "$env:USERPROFILE\.config\opencode\skills\hallucination-prevention\"
```

### विधि 3: वैश्विक नियम के रूप में (प्रत्येक सत्र में स्वचालित लोड — अनुशंसित)

```bash
# macOS / Linux
cp AGENTS.md ~/.config/opencode/AGENTS.md

# Windows PowerShell
Copy-Item AGENTS.md "$env:USERPROFILE\.config\opencode\AGENTS.md"
```

> वैश्विक फ़ाइल `AGENTS.md` प्रत्येक सत्र में स्वचालित रूप से लोड होती है, बिना मैन्युअल रूप से कौशल को बुलाए।

## उपयोग

### माँग पर (विधि 1, 2)

OpenCode स्वचालित रूप से कौशल की खोज करता है। एजेंट आवश्यकता होने पर इसे लोड करता है:

```javascript
skill({ name: "hallucination-prevention" })
```

### स्वचालित लोड (विधि 3)

`AGENTS.md` के नियम स्वचालित रूप से प्रत्येक सत्र के संदर्भ में डाले जाते हैं। **किसी मैन्युअल क्रिया की आवश्यकता नहीं है।**

## अनुमति कॉन्फ़िगरेशन

`opencode.json` में:

```json
{
  "permission": {
    "skill": {
      "hallucination-prevention": "allow"
    }
  }
}
```

विशिष्ट एजेंटों के लिए:

```json
{
  "agent": {
    "plan": {
      "permission": {
        "skill": {
          "hallucination-prevention": "allow"
        }
      }
    }
  }
}
```

अनुमतियाँ वाइल्डकार्ड पैटर्न का समर्थन करती हैं:

| अनुमति | व्यवहार |
|---------|----------|
| `allow` | कौशल तुरंत लोड होता है |
| `deny` | कौशल एजेंट से छिपा रहता है, पहुँच अस्वीकृत |
| `ask` | लोड करने से पहले पुष्टि माँगता है |

## कौशल सामग्री v2.0

### 8 प्रकार के भ्रमों का वर्गीकरण

| प्रकार | विवरण | उदाहरण | रोकथाम |
|--------|--------|---------|---------|
| **A: तथ्यात्मक** | असत्यापित तथ्य बताना | "यह फ़ंक्शन Promise लौटाता है" (कोड पढ़े बिना) | स्रोत पढ़ें, कोड चलाएँ |
| **B: संरचनात्मक** | फ़ाइलों/पथों का आविष्कार | "हैंडलर src/routes/auth.ts में है" (फ़ाइल मौजूद नहीं) | सत्यापित करने के लिए glob/ls का उपयोग करें |
| **C: व्यवहारिक** | कोड व्यवहार मान लेना | "बग ठीक हो गया" (प्रोजेक्ट चलाए बिना) | कोड चलाएँ और परिणाम दिखाएँ |
| **D: बाह्य** | API/फ़ंक्शन का आविष्कार | "Lodash में groupByAndSort() फ़ंक्शन है" (मौजूद नहीं) | दस्तावेज़ या स्रोत देखें |
| **E: परिणामात्मक** | परीक्षण परिणामों की कल्पना | "सभी परीक्षण पास होते हैं" (बिना चलाए) | परीक्षण चलाएँ, वास्तविक परिणाम दिखाएँ |
| **F: कालिक** | गलत संस्करण/तिथियाँ | "Node v24 ने यह सुविधा जोड़ी" (v24 मौजूद नहीं) | संस्करण और चेंजलॉग देखें |
| **G: आधिकारिक** | सत्यापित करने का दिखावा | "मैंने पुष्टि की कि परिवर्तन सही है" (बिना दोबारा पढ़े) | सत्यापन करें और साक्ष्य दिखाएँ |
| **H: संदर्भात्मक** | URL/ID/कमिट का आविष्कार | "देखें https://github.com/foo/issues/99999" (नकली लिंक) | URL सत्यापित करें या "सत्यापित नहीं" लिखें |

### उपकरण-विशिष्ट नियम

| उपकरण | मुख्य नियम |
|--------|------------|
| **read** | सामग्री का अनुमान न लगाएँ; फ़ाइल अस्तित्व सत्यापित करें; संपादन के बाद पुनः पढ़ें |
| **write** | लिखने के तुरंत बाद पुनः पढ़ें; बिना पूछे दस्तावेज़ न बनाएँ |
| **edit** | संपादन से पहले हमेशा पढ़ें; बाद में पुनः पढ़ें; इंडेंटेशन का सम्मान करें |
| **bash** | कमांड आउटपुट न गढ़ें; सुधार सत्यापित करने के लिए पुनः चलाएँ; विनाशकारी कार्रवाइयों से पहले पूछें |
| **grep** | grep के माध्यम से फ़ंक्शन/क्लास अस्तित्व की पुष्टि करें |
| **glob** | पथ संदर्भित करने से पहले संरचना सत्यापित करने के लिए glob का उपयोग करें |
| **question** | अस्पष्ट निर्देशों पर, पहले पूछें, फिर कार्य करें |

### संदर्भ एंकरिंग प्रोटोकॉल

1. **महत्वपूर्ण फ़ाइल संचालन से पहले**: प्रासंगिक स्रोत फ़ाइलों को पुनः पढ़ें
2. **5+ उपकरण कॉल के बाद**: संशोधित फ़ाइलों को पुनः पढ़ें, संचित स्थिति सत्यापित करें
3. **किसी कार्य पर वापस लौटते समय**: सभी प्रासंगिक फ़ाइलों को पुनः पढ़ें — स्मृति पर भरोसा न करें

### भाषा/फ्रेमवर्क-विशिष्ट नियम

- **JS/TS**: आयात करने से पहले package.json देखें; tsconfig.json देखें
- **Python**: आयात करने से पहले requirements.txt/pyproject.toml देखें; python --version देखें
- **Rust**: Cargo.toml देखें; rust-toolchain.toml देखें
- **Go**: go.mod देखें; go version देखें
- **Shell**: स्क्रिप्ट बनाने से पहले कमांड उपलब्धता देखें; सुझाव देने से पहले कमांड परीक्षण करें

### Git-विशिष्ट नियम

- कभी भी कमिट हैश न गढ़ें — `git log` का उपयोग करें
- कभी भी कमिट संदेश न मानें — `git log --oneline` का उपयोग करें
- कभी भी PR नंबर या Issue ID न गढ़ें
- कमिट के बाद, `git status` और `git log -1` चलाएँ
- ब्रांच के अस्तित्व का दावा करने से पहले, `git branch -a` चलाएँ

### विश्वास स्तर

- **सत्यापित**: मैंने सीधे पढ़ा/चलाया/सत्यापित किया
- **संभावित**: ठोस साक्ष्य लेकिन प्रत्यक्ष सत्यापन के बिना
- **अनुमानित**: समान पैटर्न पर आधारित अनुमान, सत्यापन की आवश्यकता
- **अज्ञात**: उपलब्ध जानकारी से निर्धारित नहीं किया जा सकता

डिफ़ॉल्ट रूप से, **सत्यापित** स्तर का उपयोग करें।

### एस्केलेशन प्रोटोकॉल (कब रुकें और उपयोगकर्ता से पूछें)

तुरंत रुकें और उपयोगकर्ता से पूछें जब:
1. निर्देश अस्पष्ट हों, कई व्याख्याएँ संभव हों
2. आवश्यक उपकरण/लाइब्रेरी उपलब्ध न हो
3. संचालन के लिए उच्च विशेषाधिकारों की आवश्यकता हो
4. जो देखा गया वह उपयोगकर्ता के विवरण का खंडन करता हो
5. कार्य में ऐसा ज्ञान आवश्यक हो जिसे आप सत्यापित नहीं कर सकते
6. संचालन से डेटा हानि या सुरक्षा समस्याएँ हो सकती हों
7. संदर्भ सटीक रूप से अनुसरण करने के लिए बहुत अधिक बदल गया हो
