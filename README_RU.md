# Навык OpenCode: Предотвращение галлюцинаций ИИ

> Навык агента OpenCode, который предотвращает галлюцинации ИИ путем принудительной верификации, признания неопределенности и ответов, основанных на доказательствах.

[English](./README.md) | [中文](./README_CN.md) | Русский

## Возможности

- **Проверяй, прежде чем утверждать** — читай файлы, запускай код, проверяй документацию
- **Признавай неопределённость** — чётко разделяй предположения и проверенные факты
- **Не выдумывай** — никогда не фабрикуй ошибки, API, тесты или исправления
- **Основывайся на доказательствах** — указывай пути файлов, номера строк и реальный вывод
- **Самопроверка** — 10-пунктовый чек-лист перед каждым ответом
- **Защита генерации кода** — двойная проверка до и после написания кода
- **Правила для каждого инструмента** — специфические антигаллюцинационные правила для read, write, edit, bash, grep, glob и др.
- **8 типов галлюцинаций** — каждый тип имеет свою стратегию предотвращения

## Быстрая установка

### Способ 1: Клонировать репозиторий (рекомендуется)

```bash
git clone https://github.com/magiccloudflash/opencode-skill-hallucination-prevention.git
```

Затем скопируйте навык в глобальную конфигурацию OpenCode:

```bash
# macOS / Linux
cp -r hallucination-prevention ~/.config/opencode/skills/

# Windows PowerShell
Copy-Item -Recurse hallucination-prevention "$env:USERPROFILE\.config\opencode\skills\"
```

### Способ 2: Установка вручную

```bash
# macOS / Linux
mkdir -p ~/.config/opencode/skills/hallucination-prevention
cp hallucination-prevention/SKILL.md ~/.config/opencode/skills/hallucination-prevention/

# Windows PowerShell
mkdir -p "$env:USERPROFILE\.config\opencode\skills\hallucination-prevention"
Copy-Item hallucination-prevention/SKILL.md "$env:USERPROFILE\.config\opencode\skills\hallucination-prevention\"
```

### Способ 3: Как глобальное правило (автозагрузка на каждую сессию — рекомендуется)

```bash
# macOS / Linux
cp AGENTS.md ~/.config/opencode/AGENTS.md

# Windows PowerShell
Copy-Item AGENTS.md "$env:USERPROFILE\.config\opencode\AGENTS.md"
```

> Глобальный файл `AGENTS.md` автоматически загружается в каждую сессию, не требуя ручного вызова навыка.

## Использование

### По требованию (Способ 1, 2)

Навык автоматически обнаруживается OpenCode. Агент загрузит его при необходимости:

```javascript
skill({ name: "hallucination-prevention" })
```

### Автозагрузка (Способ 3)

Правила из `AGENTS.md` автоматически внедряются в контекст каждой сессии. **Никаких ручных действий не требуется.**

## Настройка разрешений

В файле `opencode.json`:

```json
{
  "permission": {
    "skill": {
      "hallucination-prevention": "allow"
    }
  }
}
```

Для отдельных агентов:

```json
{
  "agent": {
    "plan": {
      "permission": {
        "skill": {
          "hallucination-prevention": "allow"
        }
      }
    }
  }
}
```

Режимы разрешений поддерживают шаблоны:

| Разрешение | Поведение |
|------------|-----------|
| `allow` | Навык загружается немедленно |
| `deny` | Навык скрыт от агента, доступ запрещён |
| `ask` | Запрос подтверждения перед загрузкой |

## Содержание навыка v2.0

### Классификация 8 типов галлюцинаций

| Тип | Описание | Пример | Предотвращение |
|-----|----------|--------|----------------|
| **A: Фактическая** | Утверждение непроверенных фактов | "Функция возвращает Promise" (без чтения кода) | Читай исходники, запускай код |
| **B: Структурная** | Выдумывание файлов/путей | "Обработчик в src/routes/auth.ts" (файла нет) | Используй glob/ls для проверки |
| **C: Поведенческая** | Предположение поведения кода | "Баг исправлен" (без запуска проекта) | Выполни код и покажи результат |
| **D: Внешняя** | Выдумывание API/функций | "У Lodash есть groupByAndSort()" (нет такой) | Проверь документацию или исходники |
| **E: Результативная** | Воображение тестов | "Все тесты проходят" (без запуска) | Запусти тесты, покажи реальный вывод |
| **F: Временная** | Неверные версии/даты | "Node v24 добавил эту функцию" (v24 не существует) | Проверь версию и changelog |
| **G: Авторитетная** | Притворство проверки | "Я подтвердил, что изменение верно" (без перечитывания) | Выполни проверку и покажи доказательства |
| **H: Ссылочная** | Выдумывание URL/ID/коммитов | "См. https://github.com/foo/issues/99999" (не существует) | Проверь URL или укажи "не проверено" |

### Правила для каждого инструмента

| Инструмент | Основное правило |
|------------|-----------------|
| **read** | Не угадывай содержимое; проверяй существование файла; перечитывай после редактирования |
| **write** | Сразу перечитывай написанное; не создавай документацию без запроса |
| **edit** | Всегда читай перед редактированием; перечитывай после; сохраняй отступы |
| **bash** | Не выдумывай вывод команд; перезапускай для проверки исправлений; спрашивай перед деструктивными операциями |
| **grep** | Подтверждай существование функций/классов через grep |
| **glob** | Используй glob для проверки структуры перед указанием путей |
| **question** | При неоднозначности запроса сначала спроси, потом действуй |

### 10-пунктовый чек-лист самопроверки

Перед каждым ответом проверь:

| # | Вопрос | Действие при "Нет" |
|---|--------|-------------------|
| 1 | Я действительно прочитал файлы, которые обсуждаю? | Прочитай сейчас |
| 2 | Я действительно запустил команды, о которых сообщаю? | Запусти сейчас |
| 3 | Я действительно проверил пакет/API, на который ссылаюсь? | Проверь или отметь неопределённость |
| 4 | Я уверен, что это сообщение об ошибке реально? | Покажи фактический вывод |
| 5 | Я перечитал файлы после редактирования? | Перечитай сейчас |
| 6 | Я угадываю структуру каталогов? | Выведи список каталогов |
| 7 | Я угадываю сигнатуру функции? | Прочитай исходный код |
| 8 | Я утверждаю, что исправление работает без тестирования? | Запусти код |
| 9 | Я выдумываю флаги команд? | Проверь --help |
| 10 | Мой ответ может содержать вымышленные данные? | Оставь только реальный вывод |

### Протокол привязки контекста

1. **Перед серьёзными операциями с файлами**: перечитай соответствующие исходные файлы
2. **После 5+ вызовов инструментов**: перечитай изменённые файлы, проверь накопленное состояние
3. **При возврате к задаче**: перечитай ВСЕ релевантные файлы — не полагайся на память

### Правила для языков/фреймворков

- **JS/TS**: Проверяй package.json перед импортом; проверяй tsconfig.json
- **Python**: Проверяй requirements.txt/pyproject.toml; проверь python --version
- **Rust**: Проверяй Cargo.toml; проверь rust-toolchain.toml
- **Go**: Проверяй go.mod; проверь go version
- **Shell**: Проверь доступность команд перед скриптингом; тестируй команды перед предложением

### Git-специфичные правила

- Никогда не выдумывай хеши коммитов — используй `git log`
- Никогда не предполагай сообщения коммитов — используй `git log --oneline`
- Никогда не выдумывай номера PR или Issue ID
- После коммита запусти `git status` и `git log -1`
- Прежде чем утверждать о существовании ветки, запусти `git branch -a`

### Уровни уверенности

- **Проверено**: Я непосредственно читал/запускал/проверял
- **Вероятно**: Веские доказательства, но без прямой проверки
- **Предположительно**: Вывод на основе похожих паттернов, требуется проверка
- **Неизвестно**: Невозможно определить из доступной информации

По умолчанию используй уровень **Проверено**.

### Протокол эскалации (когда остановиться и спросить пользователя)

Немедленно остановись и спроси пользователя, когда:
1. Инструкции неоднозначны, возможны разные трактовки
2. Необходимый инструмент/библиотека недоступны
3. Операция требует повышенных привилегий
4. Наблюдаемое противоречит описанию пользователя
5. Задача требует знаний, которые вы не можете проверить
6. Операция может привести к потере данных или проблемам безопасности
7. Контекст изменился слишком сильно для точного отслеживания
