# مهارت OpenCode: جلوگیری از توهم هوش مصنوعی

> مهارت عامل OpenCode که از توهم هوش مصنوعی با اعمال تأیید، پذیرش عدم قطعیت و پاسخ‌های مبتنی بر شواهد جلوگیری می‌کند.

## ویژگی‌ها

- **تأیید قبل از اظهار** — فایل‌ها را بخوانید، کد را اجرا کنید، مستندات را بررسی کنید قبل از ادعا
- **پذیرش عدم قطعیت** — حدس و گمان را از حقایق تأیید شده به وضوح جدا کنید
- **اختراع نکنید** — هرگز خطاها، APIها، نتایج تست یا رفع‌های اشکال را جعل نکنید
- **مبتنی بر شواهد** — به مسیر فایل‌ها، شماره خطوط و خروجی واقعی استناد کنید
- **جریان خودبررسی** — چک‌لیست ۵ مرحله‌ای تأیید قبل از هر پاسخ
- **محافظ‌های تولید کد** — تأیید قبل و بعد از نوشتن
- **پروتکل مدیریت خطا** — خواندن خطاهای واقعی، نمایش دقیق، تأیید رفع‌ها

## نصب

### روش ۱: کلون کردن این مخزن

```bash
git clone https://github.com/<your-org>/opencode-skill-hallucination-prevention.git
```

سپس مهارت را به تنظیمات سراسری opencode کپی کنید:

```bash
# macOS / Linux
cp -r hallucination-prevention ~/.config/opencode/skills/

# Windows PowerShell
Copy-Item -Recurse hallucination-prevention "$env:USERPROFILE\.config\opencode\skills\"
```

### روش ۲: نصب دستی

دایرکتوری مهارت را ایجاد کرده و `SKILL.md` را کپی کنید:

```bash
# macOS / Linux
mkdir -p ~/.config/opencode/skills/hallucination-prevention

# Windows PowerShell
mkdir -p "$env:USERPROFILE\.config\opencode\skills\hallucination-prevention"
```

سپس فایل [`hallucination-prevention/SKILL.md`](hallucination-prevention/SKILL.md) را در آن دایرکتوری قرار دهید.

### روش ۳: به عنوان قانون سراسری (بارگذاری خودکار در هر جلسه)

قوانین پیشرفته را به AGENTS.md سراسری خود اضافه کنید:

```bash
# macOS / Linux
cat hallucination-prevention/SKILL.md >> ~/.config/opencode/AGENTS.md

# Windows PowerShell
Add-Content -Path "$env:USERPROFILE\.config\opencode\AGENTS.md" -Value (Get-Content "hallucination-prevention\SKILL.md")
```

یا از فایل [`AGENTS.md`](AGENTS.md) مستقل موجود در این مخزن استفاده کنید.

## نحوه استفاده

مهارت به طور خودکار توسط OpenCode شناسایی می‌شود. عامل آن را در لیست مهارت‌های موجود مشاهده کرده و در زمان نیاز بارگذاری می‌کند.

برای بارگذاری صریح در طول یک جلسه:

```
عامل می‌تواند از: skill({ name: "hallucination-prevention" }) استفاده کند
```

### تنظیم مجوزها

در فایل `opencode.json` خود می‌توانید دسترسی را کنترل کنید:

```json
{
  "permission": {
    "skill": {
      "hallucination-prevention": "allow"
    }
  }
}
```

برای بازنویسی برای هر عامل:

```json
{
  "agent": {
    "plan": {
      "permission": {
        "skill": {
          "hallucination-prevention": "allow"
        }
      }
    }
  }
}
```

## محتوای مهارت v2.0

### طبقه‌بندی ۸ نوع توهم

| نوع | توضیحات | مثال | پیشگیری |
|-----|---------|-------|---------|
| **الف: واقعی** | بیان حقایق تأیید نشده | «این تابع Promise برمی‌گرداند» (بدون خواندن کد) | منبع را بخوانید، کد را اجرا کنید |
| **ب: ساختاری** | اختراع فایل‌ها/مسیرها | «هندلر در src/routes/auth.ts است» (فایل وجود ندارد) | از glob/ls برای کشف ساختار واقعی استفاده کنید |
| **ج: رفتاری** | فرض رفتار کد بدون اجرا | «باگ الان رفع شد» (بدون اجرای پروژه) | کد را اجرا کرده و خروجی واقعی را بگیرید |
| **د: خارجی** | اختراع متدهای API/کتابخانه‌ها | «Lodash تابع groupByAndSort() دارد» (ندارد) | مستندات، کد منبع یا تعاریف نوع را بررسی کنید |
| **هـ: نتیجه‌ای** | تصور نتایج تست، خروجی‌ها | «همه تست‌ها پاس می‌شوند» (بدون اجرا) | دستور واقعی را اجرا کرده و خروجی واقعی را نشان دهید |
| **و: زمانی** | ادعای نسخه‌ها/تاریخ‌های غلط | «این ویژگی در Node v24 اضافه شد» (v24 وجود ندارد) | نسخه فعلی، تغییرات یا یادداشت‌های انتشار را بررسی کنید |
| **ز: اقتداری** | تظاهر به انجام بررسی انجام نشده | «تغییر را تأیید کرده‌ام» (بدون بازخوانی فایل) | بررسی را انجام دهید و شواهد نشان دهید |
| **ح: ارجاعی** | اختراع URLها، هش commitها، شماره PRها | «به https://github.com/foo/bar/issues/99999 مراجعه کنید» (جعلی) | فقط به URLها/شناسه‌هایی که واقعاً بازدید/تأیید کرده‌اید ارجاع دهید |

### قوانین هر ابزار

| ابزار | قانون اصلی |
|-------|-----------|
| **read** | هرگز محتوای فایل را حدس نزنید. قبل از خواندن وجود فایل را تأیید کنید. پس از ویرایش دوباره بخوانید. |
| **write** | همیشه فایل نوشته شده را دوباره بخوانید. بدون درخواست مستندات ایجاد نکنید. |
| **edit** | همیشه قبل از ویرایش بخوانید. پس از ویرایش دوباره بخوانید تا تأیید شود. اگر ویرایش ناموفق بود، دوباره بخوانید تا متن واقعی پیدا شود. |
| **bash** | هرگز خروجی دستور را جعل نکنید. دستورات را اجرا کنید تا خروجی واقعی ببینید. بدون آزمایش مجدد ادعای رفع نکنید. |
| **grep** | برای تأیید وجود تابع/کلاس قبل از ارجاع از grep استفاده کنید. |
| **glob** | برای کشف ساختار واقعی فایل قبل از ارجاع به مسیرها از glob استفاده کنید. |
| **question** | وقتی دستورالعمل‌ها مبهم هستند، قبل از اقدام سؤال بپرسید. گزینه‌های مشخص ارائه دهید. |

### چک‌لیست ۱۰ نقطه‌ای خودبازبینی

| # | بررسی | اگر «نه» → |
|---|-------|-----------|
| ۱ | آیا فایل‌هایی را که درباره‌شان صحبت می‌کنم خوانده‌ام؟ | اکنون بخوانید |
| ۲ | آیا دستوراتی را که خروجی‌شان را گزارش می‌دهم اجرا کرده‌ام؟ | اکنون اجرا کنید |
| ۳ | آیا بسته/APIای را که ارجاع می‌دهم بررسی کرده‌ام؟ | بررسی کنید یا عدم قطعیت را علامت بزنید |
| ۴ | آیا مطمئنم این پیام خطا واقعی است؟ | خروجی واقعی را نشان دهید |
| ۵ | آیا فایل‌ها را پس از ویرایش دوباره خوانده‌ام؟ | اکنون دوباره بخوانید |
| ۶ | آیا ساختار دایرکتوری را حدس می‌زنم؟ | فهرست کنید |
| ۷ | آیا امضای تابعی را حدس می‌زنم؟ | کد منبع را بخوانید |
| ۸ | آیا ادعا کرده‌ام رفع‌اشکال بدون تست کار می‌کند؟ | کد را اجرا کنید |
| ۹ | آیا پرچم دستوری را جعل می‌کنم؟ | --help را بررسی کنید |
| ۱۰ | آیا پاسخ من ممکن است شامل خروجی جعلی باشد؟ | فقط به خروجی واقعی محدود کنید |

### پروتکل تثبیت زمینه

۱. **قبل از عملیات عمده فایل** — فایل‌های منبع مرتبط را دوباره بخوانید تا خود را در واقعیت فعلی تثبیت کنید.
۲. **پس از ۵+ فراخوانی ابزار** — فایل‌های اصلاح شده را دوباره بخوانید تا وضعیت انباشته شده تأیید شود.
۳. **هنگام بازگشت به یک کار** — همه فایل‌های مرتبط با کار فعلی را دوباره بخوانید. به حافظه اعتماد نکنید.

### قوانین مخصوص زبان/چارچوب

| زبان | قانون |
|-------|-------|
| **JS/TS** | قبل از import، وجود وابستگی‌ها را در package.json بررسی کنید. قبل از ادعای تنظیمات strict، tsconfig.json را بررسی کنید. |
| **Python** | قبل از import، requirements.txt/pyproject.toml را بررسی کنید. قبل از اجرا، فعال بودن محیط مجازی را تأیید کنید. |
| **Rust** | برای وابستگی‌ها و ویژگی‌ها Cargo.toml را بررسی کنید. rust-toolchain.toml را برای نسخه toolchain تأیید کنید. |
| **Go** | برای نام ماژول و وابستگی‌ها go.mod را بررسی کنید. قبل از استفاده از ویژگی‌های خاص نسخه، go version را تأیید کنید. |
| **Shell** | محیط شل (bash در مقابل zsh در مقابل powershell) را تأیید کنید. قبل از اسکریپت‌نویسی، دستورات موجود را بررسی کنید. |

### قوانین مخصوص Git

- هرگز هش commit را جعل نکنید — از `git log` برای دیدن هش‌های واقعی استفاده کنید
- هرگز پیام‌های commit را حدس نزنید — از `git log --oneline` استفاده کنید
- هرگز شماره PR یا ID issue را جعل نکنید
- پس از commit، `git status` و `git log -1` را اجرا کنید
- قبل از ادعای وجود شاخه، `git branch -a` را اجرا کنید

### سطوح اطمینان

| سطح | توضیحات |
|------|-----------|
| **تأیید شده** | مستقیماً خوانده/اجرا/بررسی کرده‌ام |
| **محتمل** | شواهد قوی اما مستقیماً تأیید نکرده‌ام |
| **حدسی** | استنباط بر اساس الگوهای مشابه، نیاز به تأیید دارد |

پیش‌فرض به **تأیید شده** — همیشه از ابزارها برای رسیدن به این سطح استفاده کنید.

### پروتکل escalation (توقف و سؤال)

بلافاصله توقف کرده و از کاربر بپرسید وقتی:

۱. **دستورالعمل‌های مبهم** — درخواست مستعد تفاسیر متعدد است
۲. **وابستگی‌های مفقود** — ابزار/کتابخانه مورد نیاز نصب نشده است
۳. **مشکلات مجوز** — عملیات نیاز به دسترسی سطح بالا دارد
۴. **وضعیت متناقض** — آنچه مشاهده می‌کنید با توصیف کاربر مغایرت دارد
۵. **مرز دانش** — کار نیاز به دانشی دارد که نمی‌توانید تأیید کنید
۶. **نگرانی امنیتی** — عملیات ممکن است باعث از دست رفتن داده یا مشکلات امنیتی شود
۷. **خستگی زمینه** — تغییرات زیادی برای ردیابی دقیق رخ داده است
